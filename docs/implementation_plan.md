# LinkedIn CV Updater FastMCP - Development Execution Plan

This document outlines the detailed development execution plan to transition the `linkedin-cv-updater` FastMCP server from a scaffolded mock to a fully functional, production-ready tool. 

The plan is divided into 7-day sprints, with day-by-day ticket breakdowns.

## User Review Required

> [!IMPORTANT]
> The unofficial `linkedin-api` library might require reverse-engineering the specific `update_experience` or `add_experience` endpoints of LinkedIn's Voyager API, as it doesn't have a built-in method for this by default. Sprint 1 dedicates time to this research and implementation. Are you comfortable with maintaining this potentially fragile unofficial API integration?

> [!WARNING]
> Using plain username/password for LinkedIn often triggers CAPTCHAs or 2FA. The plan includes a ticket to handle session cookie authentication. Do you want to prioritize browser cookie extraction early on to mitigate this?

## Sprint 1: Core API Integration & Session Management (Days 1-7)

**Goal:** Establish a real, working connection to LinkedIn that can actually create or update an experience entry on a user's profile, bypassing basic authentication hurdles.

* **Day 1: Ticket 1.1 - Research & Map Voyager API**
  * **Description:** Investigate the LinkedIn Voyager API endpoints used for adding/editing profile experiences. Document the required JSON payload (company ID, title, description, dates).
* **Day 2: Ticket 1.2 - Implement `add_experience` in `LinkedinClient`**
  * **Description:** Replace the mock `update_experience` method in `linkedin_client.py` with an actual HTTP POST/PUT request using the authenticated session from `linkedin-api`.
* **Day 3: Ticket 1.3 - Session Management & Cookie Auth**
  * **Description:** Update authentication to support `li_at` and `JSESSIONID` cookies as an alternative to username/password to avoid CAPTCHA blocks.
* **Day 4: Ticket 1.4 - Company Resolution Logic**
  * **Description:** Implement a helper to search for a company name on LinkedIn and retrieve its internal ID (URN), so the experience is linked to the correct company page rather than just text.
* **Day 5: Ticket 1.5 - Date Handling & Schema Update**
  * **Description:** Update the `ExperienceInput` schema in `update_experience.py` to accept `start_date` and `end_date` (or `is_current`). Map these to the LinkedIn payload.
* **Day 6: Ticket 1.6 - Error Handling & Retries**
  * **Description:** Implement robust error handling for API limits, expired cookies, and malformed payloads. Add retry logic for transient network errors.
* **Day 7: Ticket 1.7 - Sprint 1 End-to-End Test**
  * **Description:** Run a full manual test to add a dummy project to a test LinkedIn account and verify it appears correctly formatted on the profile.

---

## Sprint 2: FastMCP Refinement & Testing (Days 8-14)

**Goal:** Solidify the FastMCP tool interface, ensure the data generated by the Antigravity IDE is formatted beautifully, and add comprehensive unit tests.

* **Day 8: Ticket 2.1 - Bullet Point Formatting Polish**
  * **Description:** Refine the Markdown/text formatting logic in `update_linkedin_experience`. Ensure the tech stack and metrics are styled in a highly readable, professional manner.
* **Day 9: Ticket 2.2 - Implement Dry-Run Mode**
  * **Description:** Add a `--dry-run` flag or a parameter to the tool that allows the IDE to simulate the update and show the user the proposed LinkedIn text without actually mutating the profile.
* **Day 10: Ticket 2.3 - Unit Testing Setup (pytest)**
  * **Description:** Configure `pytest` and write foundational unit tests for the FastMCP tool schema validation and string formatting logic.
* **Day 11: Ticket 2.4 - Mocking LinkedIn API for Tests**
  * **Description:** Implement `unittest.mock` for `LinkedinClient` to allow testing the `update_experience` tool without making real network requests.
* **Day 12: Ticket 2.5 - Comprehensive Test Coverage**
  * **Description:** Write test cases covering edge cases (missing fields, excessively long descriptions, API failure states). Aim for 80%+ coverage in `src/`.
* **Day 13: Ticket 2.6 - Logging & Observability**
  * **Description:** Add detailed debug logging throughout the client and server. Ensure logs can be easily read from the FastMCP runtime.
* **Day 14: Ticket 2.7 - Sprint 2 Review & Code Refactor**
  * **Description:** Review the codebase for PEP8 compliance, clean up unused imports, and refactor any monolithic functions in the client.

---

## Sprint 3: Documentation, Automation, & Release (Days 15-21)

**Goal:** Make the tool easy to deploy, configure, and use for end-users, while setting up CI/CD for future maintenance.

* **Day 15: Ticket 3.1 - Browser Cookie Extraction Script**
  * **Description:** Create an optional utility script (e.g., using `browser-cookie3`) to automatically extract LinkedIn session cookies from Chrome/Firefox to populate the `.env` file effortlessly.
* **Day 16: Ticket 3.2 - Agent Prompt Engineering**
  * **Description:** Write a `system_prompt.md` or instructional guide that defines exactly how the Antigravity IDE should extract metrics and technologies to feed into this FastMCP tool.
* **Day 17: Ticket 3.3 - Update README & Setup Guides**
  * **Description:** Completely overhaul the `README.md` with detailed instructions on setting up cookies, running the server, and configuring the IDE integration.
* **Day 18: Ticket 3.4 - Continuous Integration (GitHub Actions)**
  * **Description:** Set up a `.github/workflows/test.yml` file to automatically run `pytest` and linting on every push or pull request.
* **Day 19: Ticket 3.5 - Dependency Pinning & Security Scan**
  * **Description:** Pin exact dependency versions in `pyproject.toml` / `uv.lock`. Run a vulnerability scan on dependencies.
* **Day 20: Ticket 3.6 - Release Tagging & Packaging**
  * **Description:** Prepare the v1.0.0 release. Build the wheel and source distributions using `hatchling`.
* **Day 21: Ticket 3.7 - Final Polish & Demo**
  * **Description:** Record a demonstration of the IDE pulling context and updating LinkedIn. Complete final manual verification.

## Verification Plan

### Automated Tests
- `pytest tests/` will be run to ensure schema validation and formatting logic works as expected.
- CI pipeline will enforce passing tests on PRs.

### Manual Verification
- A test LinkedIn account will be used to verify that the `LinkedinClient` successfully adds and updates an Experience item on the profile.
- We will trigger the tool directly from the Antigravity IDE to ensure the MCP communication is fully functional.
